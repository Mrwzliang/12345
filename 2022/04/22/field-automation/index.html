field_automation

发表于 2022-04-22 分类于 源码

基础概念：
总所周知，field_automation主要是封装了一些常用的方法比如：打印，比较等，这样便于代码的简洁。

注意点：
1.派生关系：自定义的类(my_transaction)-》uvm_sequence_item-》uvm_transaction-》uvm_object.
my_transaction是整个验证平台中流动的信息单元，是穿梭在验证平台中的子弹。
2.自定义的transaction中每个成员变量前都有rand修饰符，这样在调用此类的randomize方法时才会给相应的字段一个随机值。
3.定义格式：uvm_object_utils_begin(mac_transaction).... uvm_object_utils_end,中间定义具体的字段即属性，
此与factory中的uvm_object_utils类似，其实field_automation也是基于工厂机制实现的。
为什么要这样定义呢？我想可能是UVM是面向对象语言所描述，那么万物皆是对象，自然一个object用fatory来实现很合理~
4.关于uvm_field_xxx(xxx)宏有很多，xxx:包括一些数据类型：int,real,enum,string,array,queued等，
参数：name：trans’s name,属性：uvm_nopack等，enum还有个T表示enum类型的名字。

场景举例：
假设我们的mac_transaction中多了一个crc_err的字段，这个字段用于控制向激励中施加crc错误，我们可以控制这个标志位在验证
平台上流动但是又不发给driver。uvm_field_int(crc_err, UVM_ALL_ON | UVM_NOPACK),使用上述语句后，那么当pack和unpack的时候， UVM就不会考虑这个字段了。 假如现在扩充了一个新的标志：sfd_err,并在trans里定义默认的约束：constraint default_cons{ crc_err == 0;sfd_err==0;}， 这样的问题就是随机化这两个值永远都是0，但是我们期望在正常情况下为0，在测试err的时候是可0可1。做如下修改： 在正常case时，可以使用如下的方式随机化：mac_transaction tr; tr = new();assert(tr.randomize()); 在需要注入error的case时，可以使用如下的方式随机化： mac_transaction tr; tr::constraint_mode(0); tr=new();assert(tr.randomize().with{crc_err:dist{1:=2,0:=1};sfd_err:dist{1:=2,0:=1}}); 上面使用constraint_mode(0)关闭了default_cons。之后在随机化时可以指定新的约束条件。注意不关闭会报错哦。（约束冲突）这里并不是新约束覆盖旧约束的方式。 另外一种是tr.default_cons::constraint_mode(0)，这样会关闭某个特定的约束。在上面的例子中，只有一个约束，所以两者的效果是一样的。 假如有100个标志位，那每次随机的时候要写100个位，所以尽量在trans里定义的时候，拆分多个约束，每个约束里一个或多个标志位就行了。 也可以在宏里面使用if条件有选择的加入某些字段。 rand bit is_vlan; uvm_object_utils_begin(mac_transaction)
uvm_field_int(smac, UVM_ALL_ON) uvm_field_int(dmac, UVM_ALL_ON)
if(is_vlan) begin
uvm_field_int(vlan_info1, UVM_ALL_ON) uvm_field_int(vlan_info2, UVM_ALL_ON)
uvm_field_int(vlan_info3, UVM_ALL_ON) uvm_field_int(vlan_info4, UVM_ALL_ON)
end
uvm_field_int(eth_type, UVM_ALL_ON) uvm_field_array_int(pload, UVM_ALL_ON)
uvm_field_int(crc, UVM_ALL_ON) uvm_object_utils_end
然后在外面随机化时这样选择：mac_transaction tr;tr = new();assert(tr.randomize() with {is_vlan == 0/1;})

剖析源码：
主要从某个点去追一下代码，其他点也相差不大。
首先看一下下面这段代码的基本结构及来源。
class my_cfg extends uvm_object;
rand bit [7:0] num;
uvm_object_utils_begin(my_cfg) uvm_field_int(num, UVM_ALL_ON)
uvm_object_utils_end endclass uvm_object_utils_begin展开： define uvm_object_utils_begin(T) m_uvm_object_registry_internal(T,T)
m_uvm_object_create_func(T) m_uvm_get_type_name_func(T)
uvm_field_utils_begin(T) 这几个展开的宏中，除了uvm_field_utils_begin之外，其它的几个在factory机制时已经分析过。 对于uvm_field_utils_begin宏，其实在uvm_object_utils之中也出现过： define uvm_object_utils(T)
uvm_object_utils_begin(T) uvm_object_utils_end
显然，uvm_object_utils的展开中也是包括了uvm_field_utils_begin宏的。所以uvm_field_int是插在uvm_field_utils_begin与uvm_object_utils_end之间的，
无论uvm_field_int宏是否出现，当使用factory机制时，uvm_field_utils_begin是一定存在的。

首先是uvm_field_utils_begin,下面就总结一些内容，参考代码在src/macros/uvm_object_defines.svh路径。
里面的核心方法：
`define uvm_field_utils_begin(T)
m_uvm_field_automation((uvm_object tmp_data, int what__, string str__)
我们用上面的my_cfg来代替T，而在这个函数里面通过super的方式重载了这个方法，这个方法在uvm_object里是个空方法，单从此处无法看出作用，
正常到这我们就必须看其他方法有没有调用这个方法，自然关于field_automation,copy方法很经典，所以查看copy方法是否涉及这个方法，
在uvm_object中的copy方法：果然在copy方法里调用了此方法， __m_uvm_field_automation(rhs, UVM_COPY, “”);这里只是写了部分关键代码，
这个rhs就是copy(uvm_object rhs) 的参数，即要copy的对象的指针，第二个参数用于表明这是一个copy操作，自然field_automation有好几个操作，第三个参数
是一个空的字符串，关于这个空字符串的应用在set_x_local函数里，这里的x有int,string,object.那么这个set_x_local到底干啥用的呢？
我们以set_int_local为例看一下：首先这个函数呢也在uvm_object里，暂且不贴代码了，我们惊奇的发现在这个方法里也调用了
__m_uvm_field_automation(null, UVM_SETINT, field_name);方法，而这里是uvm_setint操作，而field_name就是设置为要设置的变量的名字。
好了到这里可能还有疑惑，这个set_x_local函数到底干嘛的呢？表面看是将变量设置为local类型，可为什么要设置为local类型呢？回头可以再深究一下下~

下面再看下操作类型，小嘬一口，点到为止。
such as:
`uvm_field_int(a, UVM_ALL_ON | UVM_NOCOPY)
定位到uvm_object_global.svh文件中：
代码就不全贴了，可以总结出：
parameter UVM_DEFAULT = ‘b000_0101_0101_0101;
parameter UVM_ALL_ON = ‘b000_0001_0101_0101;
parameter UVM_FLAGS_ON = ‘b000000101010101;
parameter UVM_FLAGS_OFF = 0;
parameter UVM_COPY = (1<<0);
parameter UVM_NOCOPY = (1<<1);
parameter UVM_COMPARE = (1<<2);
parameter UVM_NOCOMPARE = (1<<3);
parameter UVM_PRINT = (1<<4);
parameter UVM_NOPRINT = (1<<5);
所谓的UVM_ALL_ON，UVM_NOCOPY等都是一些数字，这些数字的低10位用于表明操作的类型。这里一共有五种操作：copy，compare，print，record和pack。
通过查看标志位的情况就可以知道是否执行某种操作。如对于例子中的UVM_ALL_ON | UVM_NOCOPY，
其值为’b000_0001_0101_0111，在执行copy操作时，先看一下倒数第二位，发现此位被置起了，于是就不对此字段执行copy操作。

到这里我们就可以总结一下`uvm_field_int这个宏
代码在路径：src/macros/uvm_object_defines.svh
首先声明一下这个宏结包含了多个操作方法的过程，但是相差不大，看一下copy:
假设是copy的操作之下，在执行copy函数还是会去调用__m_uvm_field_automation函数，在这里面对传进来的参数进行校验，在这里就是检验类型是否一致，
不然不可互相copy,再者通过与运算判断当前是否copy.

__m_uvm_status_container
考虑uvm_field_int宏中与compare相关的部分,这里有个关键的变量__m_uvm_status_container，首先它是uvm_object的一个成员变量，其次还是静态的，uvm_status_container类型的成员变量。
uvm_status_container的定义位于uvm_misc.svh文件中，uvm_status_container中核心的对象：uvm_scope_stack，这个类中的成员变量都是字符串类型的，用于存放UVM中的路径，并提供了一些函数来对这些路径进行存取操作。
其中比较重要的是get，down，up，down_element及up_element。

准备以上背景开始看下compare操作：
function bit uvm_object::compare (uvm_object rhs,uvm_comparer comparer=null);
刚刚得知uvm_status_container是静态的，那成员变量__m_uvm_status_container静态的，comparer指的是一个全局的comparer，因为此为uvm_status_container中的成员变量。
